### ğŸ§  **Scaling Up: Splitting Routes**

**Why split routes into separate files?**

- **Maintainability** â€“ Keeping everything in `app.js` gets messy fast.
    
- **Modularity** â€“ You want separation of concerns: admin routes vs user/shop routes, etc.
    

So you create:

- `routes/admin.js`
    
- `routes/shop.js`
    

Each exports a `router` using:

```js
const express = require('express');
const router = express.Router(); // Mini express app

// Routes
router.get('/add-product', handler);
router.post('/product', handler);

// Export
module.exports = router;
```

Then in `app.js`, you import and mount them:

```js
const adminRoutes = require('./routes/admin');
const shopRoutes = require('./routes/shop');

app.use('/admin', adminRoutes);
app.use(shopRoutes); // Mounts at root level
```

Now:

- A GET request to `/admin/add-product` triggers the `admin.js` route.
    
- A GET to `/` triggers the `shop.js` route.

Routing in Express is **first match wins**.

So:

```js

app.use(shopRoutes);
```

This ensures admin routes are prioritized over general shop routes.

But if you were to switch to `app.use()` instead of `app.get()`, and you place the `shopRoutes` _above_ the `adminRoutes`, you might unintentionally hijack admin route traffic. That's why even if it works now with `app.get()`â€”youâ€™re **better off ordering it properly from the beginning**.
